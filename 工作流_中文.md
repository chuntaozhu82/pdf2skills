下面直接给你一套「可以照着做」的方案：先是从 PDF → skills → 项目 → 设计文档 → 代码的完整 pipeline，然后是一整套你可以在 Claude 里直接复用/改造的对话模版。

## 一、从 PDF 到 skills：怎么准备 PDF

目标：让 pdf2skills 更容易把书拆成「可执行的游戏设计技能」，而不是只做摘要。

### 1.1 章节 / 目录结构建议

尽量让书的结构自然映射到你想要的 8 阶段流程：

> 1. 概念定义  
> 2. 核心循环  
> 3. 机制与平衡  
> 4. 关卡与体验  
> 5. 视觉与交互  
> 6. 玩家心理  
> 7. 原型与测试  
> 8. 团队管理  

可以按类似结构写 / 整理你的 PDF（可以是你自写的设计手册，而不一定是正式出版物）：

```text
《XXXX 游戏设计手册（PDF）》
├─ 第 1 篇 概念与定位（对应：概念定义）
│  ├─ 第 1 章 核心概念与愿景
│  └─ 第 2 章 目标玩家与平台
├─ 第 2 篇 核心循环与玩法结构（对应：核心循环）
│  ├─ 第 3 章 核心循环拆解
│  └─ 第 4 章 资源与动作系统
├─ 第 3 篇 数值机制与难度平衡（对应：机制与平衡）
│  ├─ 第 5 章 战斗/经济/成长数值
│  └─ 第 6 章 动态难度调整（DDA）
├─ 第 4 篇 关卡与节奏（对应：关卡与体验）
│  ├─ 第 7 章 关卡结构与节奏曲线
│  └─ 第 8 章 新手关卡与高难度关卡
├─ 第 5 篇 视觉与交互（对应：视觉与交互）
│  ├─ 第 9 章 UI 布局与信息层级
│  └─ 第 10 章 操作反馈与可用性
├─ 第 6 篇 玩家心理与动机（对应：玩家心理）
│  ├─ 第 11 章 动机与爽感设计
│  └─ 第 12 章 挫折点与留存
├─ 第 7 篇 原型与测试（对应：原型与测试）
│  ├─ 第 13 章 原型范围与迭代
│  └─ 第 14 章 测试指标与数据
├─ 第 8 篇 团队与流程（对应：团队管理）
│  ├─ 第 15 章 任务拆解与工期
│  └─ 第 16 章 跨职能协作
└─ 附录：表格模板、公式清单、案例
```

要点：  
- 每一章尽量就是一个潜在「子 skill」；  
- 每一篇自然聚合成「阶段 skill」，方便 pdf2skills 生成「8 阶段流程」。

### 1.2 写作风格与格式：尽量「技能化」

你希望 pdf2skills 输出的是「可执行步骤」，所以写文档时提前按「技能」的形态来写。

**每个关键小节尽量写成类似：**

```text
【技能名称】动态难度调整（DDA）

【适用场景】
- 玩家在若干关卡内失败率过高或过低
- 新手区流失高，需要保护机制

【输入】
- 玩家表现数据：通关时长、失败次数、剩余生命等
- 当前关卡难度参数：敌人生命、攻击、数量
- 目标难度区间：通关率 40%~60%

【步骤】
1. 计算玩家表现评分 P ∈ [0,1]
   P = 0.5 * 通关率 + 0.3 * 平均剩余生命 + 0.2 * 平均用时系数
2. 若 P < 0.4，则：
   - 敌人生命 × 0.8
   - 掉落回复道具概率 +15%
3. 若 P > 0.7，则：
   - 敌人攻击 × 1.1
   - 奖励资源减少 10%
4. 将调整参数记录到关卡配置表中

【输出】
- 新的关卡配置参数表
- 推荐测试用例 3 个（含输入表现、输出难度系数）
```

通用写作规则：

- **多用结构块**：`【输入】【步骤】【输出】【适用场景】【注意事项】`  
- **避免废话**：少写「比如」「其实」「大概」，多写「如果 A，则 B」  
- **用表格承载配置**：数值、曲线、概率用表格写清楚，方便 AI 直接转为 JSON  
- **一个技能一页或一小节**：不要在同一节混太多不相关内容  

这样 pdf2skills 把这些小节拆成子 skill 时，会直接把「输入/输出/步骤」映射到 skill 文件结构里，你后面在 Claude Code 调用就会非常流畅。

## 二、拿到 skills.zip 后：项目目录怎么组织

假设你上传 PDF 后拿到了 `skills.zip`，解压后建议在 Claude Code 项目里这样组织：

```text
my-pdf2game-project/
├─ .claude/
│  ├─ skills/
│  │  └─ game-design-from-book/
│  │     ├─ SKILL.md              # 主 skill：定义 8 阶段流程 + 总入口
│  │     ├─ index.md              # 索引 / 路由
│  │     └─ skills/               # 子 skills（pdf2skills 生成）
│  │        ├─ concept-definition.md
│  │        ├─ core-loop.md
│  │        ├─ difficulty-balance.md
│  │        ├─ level-design.md
│  │        ├─ ux-visual.md
│  │        ├─ player-psychology.md
│  │        ├─ prototyping-testing.md
│  │        ├─ team-management.md
│  │        └─ ... 其他更细颗粒的技能
│  └─ config/
│     └─ agents.md                 # 可选：你自定义的 Claude 行为规则
├─ design-docs/
│  ├─ game-design-doc-v0.md        # 初稿：8 阶段第一次跑出来
│  ├─ game-design-doc-v1.md        # 修改版
│  └─ gdd-locked.md                # 确认冻结版，用于代码生成
├─ src/
│  ├─ core/                        # 核心玩法逻辑
│  │  ├─ game-loop.ts
│  │  ├─ combat.ts
│  │  └─ config.ts
│  ├─ ui/                          # UI/视觉与交互
│  │  ├─ ui-layout.ts
│  │  └─ styles.css
│  ├─ levels/                      # 关卡/资源
│  │  ├─ level-data.json
│  │  └─ events.json
│  ├─ assets/                      # 图片/音频占位（可由 AI 生成描述）
│  ├─ main.ts
│  └─ index.html
└─ README.md
```

组织原则：

- **所有 pdf2skills 产物** 都放 `.claude/skills/<某个子目录>` 下，不污染业务代码目录；  
- **设计文档是单独目录**（`design-docs/`），AI 输出的每个阶段内容都落盘，方便你回滚和对比；  
- **src 下按「模块」划分**：core / ui / levels / data，对应你后面让 Claude「分模块生成代码」的步骤。

## 三、在 Claude Code 里：让它按「8 阶段流程」生成设计文档

核心思路：

1. 用一条「总控指令」告诉 Claude：有一套 game-design skills，要求它**严格按 8 阶段**来跑；  
2. 每一阶段只做一件事，输出写入 `design-docs/xxx` 文件；  
3. 阶段之间串联关系：后面的指令显式引用前一阶段的输出内容。

### 3.1 首次开启项目时的总控提示

你在 Claude Code 里打开项目后，先发一条类似这样的系统级长指令（可以存成 `.claude/config/init.md`，每次项目打开就粘）：

```markdown
你当前在一个名为 "my-pdf2game-project" 的项目中。

在 `.claude/skills/game-design-from-book` 目录下，已经安装了一套由 pdf2skills 从游戏设计书自动生成的技能包：
- SKILL.md 定义了完整的 8 阶段游戏设计流程
- skills/*.md 是子技能（核心循环、难度平衡、关卡设计、玩家心理等）

接下来，我希望你：
1. **严格按照这套技能定义的 8 阶段流程**，从 0 开始帮我完成一个具体游戏的设计文档（GDD）
2. 每一阶段结束时，将当前阶段的结构化结果写入 `design-docs/game-design-doc-v0.md` 中，对应章节
3. 在 GDD 相对稳定后，再根据 GDD 按模块生成代码（先核心玩法，再 UI，再关卡资源）

请先阅读 SKILL.md 和 skills 子目录，理解可用的阶段与子技能，然后只回答一句：
> "已加载 game-design-from-book 技能包，等待你指定第 1 个阶段的目标游戏类型。"
```

### 3.2 让它从第 1 阶段开始（概念定义）

接下来，你开始「驱动」流程。第 1 条任务指令示例：

```markdown
【阶段 1：概念定义】

目标：我想做一个「轻度 Roguelike + 卡牌构筑」的网页游戏，风格偏东方幻想，预计游戏时长 20~40 分钟。

请在调用 `concept-definition` 相关技能的前提下，完成以下输出，并写入 `design-docs/game-design-doc-v0.md` 的「第 1 章 概念与定位」：

1. 用不超过 20 个字给出一句 **核心命题**（例如：“在不断轮回中，通过牌组重塑命运”）。
2. 给出 1 段不超过 150 字的 **游戏简介**。
3. 定义目标玩家画像，包括：
   - 年龄段
   - 平台偏好（PC 网页）
   - 玩游戏的动机（如策略深度、构筑乐趣、轻度叙事等）
4. 输出 3 个「不是本游戏」的反例，用来保护边界（例如：不是重氪抽卡手游、不是高操作 ACT）。

要求：
- 请显式标明这是「阶段 1：概念定义」的输出
- 使用 Markdown 标题（`## 1. 概念与定位`）组织内容
- 结束时给出一个简要小结，说明后续阶段（核心循环设计）会基于哪些要点继续展开
```

你可以在 Claude 的回答里拷贝合适部分，或直接让它用「编辑文件」能力写入 `design-docs/game-design-doc-v0.md`。

### 3.3 后续阶段的驱动模板（2–7 阶段）

以下是每个阶段可以用的「指令骨架」，你只需要根据自己游戏改一改即可。

**阶段 2：核心循环**

```markdown
【阶段 2：核心循环】

请基于 `design-docs/game-design-doc-v0.md` 中「第 1 章 概念与定位」的内容，
调用 `core-loop` 或等价子技能，完成核心循环设计，并更新同一设计文档中的「第 2 章 核心循环」。

输出内容包括：
1. 一段不超过 100 字的文字，概述核心循环。
2. 使用 mermaid 流程图语法描述核心循环（步骤不超过 5 个节点）。
3. 用表格列出循环中涉及的关键资源与状态，例如：
   - 资源名称（如：手牌、能量、生命）
   - 获取方式
   - 消耗方式
4. 指出循环中的 2~3 个潜在风险点（例如：是否容易形成单一最优解，是否会过于 grind）。

格式要求：
- 使用 `## 2. 核心循环` 作为章节标题
- 所有 mermaid 代码块用 ```mermaid 包裹
- 明确引用上一阶段的「核心命题」

完成后请简短说明：下一阶段（机制与平衡）会如何基于当前循环展开。
```

**阶段 3：机制与平衡**

```markdown
【阶段 3：机制与平衡】

请基于「第 2 章 核心循环」内容，调用与数值和平衡相关的技能（例如 `difficulty-balance`、`reinforcement-feedback` 等），更新「第 3 章 机制与平衡」。

输出包括：
1. 核心战斗或主要互动的「伤害/收益」公式（用接近代码的伪代码表示）。
2. 玩家成长曲线示意（用表格列出若干等级的核心属性值）。
3. 难度曲线的大致规划（早期/中期/后期三段，每段给出目标体验和关键参数特征）。
4. 至少 2 个用于测试平衡性的「场景用例」（包括输入设定、期望结果）。

格式要求：
- `## 3. 机制与平衡`
- 所有公式尽量接近将来要实现的编程语言（如 TypeScript 风格）
- 在输出结尾，用一句话提醒后续「关卡与体验」阶段需要关注的输入（例如：玩家成长曲线、难度区间）
```

**阶段 4：关卡与体验**

```markdown
【阶段 4：关卡与体验】

请基于当前设计文档（特别是核心循环与难度曲线），调用 `level-design` 相关技能，更新「第 4 章 关卡与体验」。

输出包括：
1. 用表格形式列出 10 个关卡（或 10 个 run 阶段）的设计概要：
   - 关卡编号
   - 目标体验（如：新手教学、第一次高潮等）
   - 关键机制引入
   - 目标难度系数
2. 为新手前 3 个关卡写出更加细化的说明，包括：
   - 该关卡教学的知识点/机制
   - 控制失败成本的方法
3. 给出一条整体「节奏曲线」文字描述（例如：从轻松探索 → 紧张决策 → 高压冲刺 → 结局发泄）。

格式要求：
- `## 4. 关卡与体验`
- 明确说明这些数据今后将映射为 `src/levels/level-data.json`
```

**阶段 5：视觉与交互**

```markdown
【阶段 5：视觉与交互】

请调用与 UI/UX 设计相关的技能（例如 `visual-design`、`fitts-law-ui` 等），更新「第 5 章 视觉与交互」。

输出包括：
1. 主要界面列表（如：主菜单、地图界面、战斗界面、牌组管理界面），每个界面一句话描述其核心功能。
2. 每个界面提供一个「文本版线框图」（描述各 UI 区块的位置和层级）。
3. 为核心交互路径（如：抽卡→出牌→结算）列出详细步骤和反馈方式（视觉/音效/动效）。
4. 列一个 UI 组件清单，后面会映射到 `src/ui/` 中的组件（例如：Button、CardView、HealthBar 等）。

格式要求：
- `## 5. 视觉与交互`
- 所有线框图用纯文字描述，不画图，但结构清晰
```

**阶段 6：玩家心理**

```markdown
【阶段 6：玩家心理】

请调用 `player-psychology` 等相关技能，更新「第 6 章 玩家心理与动机」。

输出包括：
1. 描述本游戏的核心动机结构：收集、成长、挑战、叙事等各占比重。
2. 列出 3 个「主要爽点」和 3 个「潜在挫折点」。
3. 说明你打算如何用数值、关卡设计、奖励频率来支撑“爽点”，以及如何用 DDA 或其他方法缓冲“挫折点”。
4. 如果将来收集数据（如关卡通过率、平均在线时长），应该关注哪些指标来判断这些设计是否奏效。

格式要求：
- `## 6. 玩家心理`
- 尽量引用前面章节中的具体设计（不要空泛）
```

**阶段 7：原型与测试**

```markdown
【阶段 7：原型与测试】

请调用 `prototyping-testing` 等相关技能，更新「第 7 章 原型与测试」。

输出包括：
1. 列出首个可玩原型的范围（哪些系统必须实现，哪些可以用假数据/占位符）。
2. 规划 3 轮迭代，每一轮：
   - 要验证的关键问题
   - 需要收集的数据
   - 是否需要改动核心循环或仅改动数值/UI
3. 为首个可玩版本写一个「任务清单」（后续会转换成代码 TODO）。

格式要求：
- `## 7. 原型与测试`
- 明确指出代码生成的优先级：先核心玩法，再 UI，再关卡资源
```

**阶段 8：团队管理** 如果你只是做实验，可以简化为：用来生成一份「开发任务分解清单」而不是正式团队结构。

## 四、设计文档稳定后：如何按模块生成代码

当你认为 `design-docs/gdd-locked.md` 已经稳定，可以这样引导 Claude：

### 4.1 锁定文档版本

先让它整理一份「最终版」GDD：

```markdown
请阅读 `design-docs/game-design-doc-v0.md`，在与我快速确认后，将修改完善版写入：

- `design-docs/gdd-locked.md`

要求：
1. 章节结构保持「1~7 章」不变。
2. 删除明显重复、前后矛盾的描述。
3. 在文末增加一节「面向代码的摘要」，用列表形式总结：
   - 必须实现的模块
   - 关键配置表
   - 不可违背的设计约束（如节奏、难度区间）
```

确认后，就可以开始模块化代码生成。

### 4.2 核心玩法模块（先做 core）

```markdown
【代码阶段 1：核心玩法模块】

请仅基于 `design-docs/gdd-locked.md` 的第 2 章「核心循环」和第 3 章「机制与平衡」，并结合对应 skills（核心循环、难度平衡相关子 skill），生成 `src/core/` 下的初版代码结构。

目标语言：TypeScript（运行环境：浏览器，打包工具可暂不考虑）

要求：
1. 在 `src/core/` 下生成：
   - `game-loop.ts`：包含游戏主循环、回合流程控制
   - `combat.ts`（如果有战斗）：实现核心公式的伪代码或 TS 真实代码
   - `config.ts`：集中存放从 GDD 中抽取的参数（如基础生命、攻击、关卡参数等）
2. 每个文件开头加一行注释，标明来自哪个 skill，例如：  
   `// AI-Generated using core-loop & difficulty-balance skills`
3. 所有关键数值和公式必须能在 GDD 中追溯到出处（请在注释中引用章节号，例如：`// 来源：GDD 第 3 章 机制与平衡`）。

请先列出你将创建的文件清单和每个文件的大纲结构，再开始填充具体代码。
```

### 4.3 UI 模块（再做 ui）

```markdown
【代码阶段 2：UI / 视觉与交互】

在完成 `src/core/` 基础之后，请基于 GDD 第 5 章「视觉与交互」和相关 UI skills，生成：

- `src/ui/ui-layout.ts`：包括几个主要界面的组件结构
- `src/ui/styles.css`：定义基础配色、字体、卡牌与按钮样式
- 必要的 UI 组件，例如：`Button`, `CardView`, `HealthBar` 等

要求：
1. 对每一个 UI 组件，在源文件注释里写清：  
   - 它对应哪一节 GDD 的哪部分（例如：GDD 5.2 主界面线框图）
2. 样式可以简化，但要体现出「可玩原型」级别的清晰度（不同状态、hover/pressed 等可区分）。
3. 不要一次把所有细节写死，允许使用 TODO 标记后续优化点。
```

### 4.4 关卡和资源（最后做 levels）

```markdown
【代码阶段 3：关卡与资源配置】

### 5.4 调试 / 迭代时的对话模版

当你发现问题时，用这样的句式：

```markdown

- 敌人平均造成伤害：XX
- 玩家平均剩余生命：X%

3. 给出 1 套新的难度调整方案，并用伪代码/TS 形式更新 `calculateDifficulty()` 或对应逻辑。

请分两步做：
- 第一步：分析 + 新方案
- 第二步（我确认后）：给出具体代码 diff。
```
